Структура Баг репорта.

Шапка

1.Короткое описание (Summary)- описание проблемы явно указывающее на причину и тип ошибочной ситуации
2.Проект (Project) -Название тестируемого проекта
3.Компонент приложения (Component)- Название части  или функции тестируемого продукта
4.Номер версии (Version)- Версия на которой была найдена  ошибка
5.Серьезность (Severity)-Пятиуровневая система градации серьезности дефекта:
                          *S1 -Блокирующая (Blocked)
                          *S2 -Критическая (Critical)
                          *S3 -Значительный (Major)
                          *S4 -Незначительный (Minor)
                          *S5 -Тривиальный (Trivial)

6.Приоритет (Priority) -Приоритеты дефекта :
                        P1 -Высокий (High)
                        P2 -Средний (Medium)
                        P3 -Низкий (Low)

7.Статус (Status) - Статус бага зависит от используемой процедуры и жизненного цикла бага 
8.Автор  (Author) -Создатель баг репорта
9.Назначение (Assigned to) - Имя сотрудника,назначенного на решение проблемы

10.Окружение -ОС/Браузер+версия/ Информация об окружении,на котором был найден баг
11.Описание: 
     *Шаги воспроизведения (Steps to Reproduce) -Щаги по которым легко воспроизвести ситуацию,которая привела к ошибке
     *Фактический результат (Result) -Результат, полученный после прохождения шагов
     *Ожидаемый результат (Expected Result) Ожидаемый правильный результат

12.Дополнения (Attachment) - Файл с логами/скрин/любой документ который может прояснить ситуацию.
